1. نموذج العمل العام

كل ميزة في PMS تتبع نفس النمط:

User Action (React Page) → apiFetch + Bearer Token → Laravel Route + Middleware → Controller (validation) → Service (business logic) → Model / DB / External API → JSON Response → React State Update → UI
  • Frontend: صفحات React + Context (AuthContext, LanguageContext) + MUI.
  • Backend: Controllers رفيعة + Services للمنطق + Eloquent Models.
  • الأمان: Sanctum → EnsureUserHasUniversity → user.status → role middleware → ProjectAccess.

2. المصادقة والجلسات — كيف استخدمناها

من وجهة المستخدم

المستخدم يسجّل أو يدخل من Login.jsx / Register.jsx. بعد النجاح يُخزَّن token في localStorage ويُحمَّل الملف الشخصي.

كيف نُفِّذ

الخطوةالتفاصيل
التسجيلAuthService::register() — ينشئ user pending · يُصدر Sanctum token · يُبلّغ admins
الدخولAuthService::login() — يتحقق من credentials · يُرجع token + user + role
الجلسةAuthContext يحفظ token · عند كل طلب يرسل Authorization: Bearer
تحميل الحالةGET /profile → يحدّث status, role, university في Context
حماية الصفحاتProtectedRoute — pending/rejected → PendingApproval · active → Dashboard
GitHubOAuth stateless عبر Socialite · token يُحفظ في users.github_token
لماذا Sanctum؟ SPA تحتاج API stateless — token أسهل من session cookies للـ React المنفصل عن Laravel.

3. Multi-Tenancy — كيف استخدمناها

استخدمنا Shared Database — جامعة واحدة = tenant. كل سجل tenant-scoped له university_id.

الآلية

  1. Trait BelongsToUniversity على Models (Project, Task, …).
  2. TenantScope global scope — يفلتر تلقائياً حسب auth()->user()->university_id.
  3. Super Admin يتجاوز TenantScope — يرى كل الجامعات.
  4. عند الإنشاء: يُملأ university_id من المستخدم أو من المشروع الأب.
النتيجة: طالب جامعة A لا يرى مشاريع جامعة B — بدون تغيير في كل query يدوياً.

4. موافقة الحسابات — كيف استخدمناها

التسجيل الذاتي

  1. طالب/مشرف يسجّل → status = pending.
  2. مشرف: attachSupervisorUniversitiesPending() — pivot pending لكل جامعة.
  3. ProtectedRoute يمنع Dashboard — يعرض PendingApproval فقط.
  4. Routes Group A (profile, logout) مسموحة حتى pending.

قرار المدير

UserController::approve/reject → للمشرف يحدّث pivot → refreshAccountStatusFromMemberships():

  • أي membership active → user active
  • الكل rejected → user rejected → زر re-register
  • باقي pending → يبقى pending

إنشاء من المدير

UserController::store — ينشئ user active مباشرة (بدون workflow موافقة).

6. الدعوات — كيف استخدمناها

دعوة مشرف

  1. المالك يفتح Invites tab → يستعرض GET /supervisors.
  2. يرسل invite-supervisor → سجل في supervisor_invitations.
  3. المشرف يرى SupervisorInvitations.jsx → accept/reject.
  4. عند القبول: projects.supervisor_id يُحدَّث.

دعوة طالب

  1. المالك يستعرض طلاباً متاحين (StudentService — يستبعد الأعضاء الحاليين).
  2. دعوة → student_invitations · إشعار للطالب.
  3. قبول → project_members.status = accepted → يحصل على ProjectAccess.

7. المهام — كيف استخدمناها

UI

TasksTab.jsx — عرض · إنشاء · تغيير حالة (pending/in_progress/completed) · حذف.

Backend — TaskService

  1. createTask — ينشئ task · يسجّل ProjectActivity · notifyProjectParticipants.
  2. updateTask — إذا تغيّرت الحالة → activity + notification.
  3. deleteTask — يتحقق owner/supervisor/member · يُعيد حساب progress.

كل task يرث university_id من المشروع عبر BelongsToUniversity boot.

8. التعليقات — كيف استخدمناها

استخدمنا تعليقات على مستوى المشروع فقط — نموذج team chat.

التدفق

  1. CommentsTab.jsx — يرسل نص إلى POST /project/{id}/comment.
  2. CommentService::addProjectComment — يحفظ في comments مع project_id.
  3. العرض: GET /project/{id}/comments — ترتيب زمني chat-style.
  4. تعديل/حذف: PUT/DELETE /comment/{id} — للمالك أو صاحب التعليق.
لم نستخدم task comments في الواجهة — نقاش الفريق مركز على المشروع ككل.

9. الإصدارات و GitHub — كيف استخدمناها

Releases tab — VersionsTab.jsx

الفعلكيف
رفعFormData → ProjectVersionService::uploadVersion → file في storage/public/versions
قائمةGET /project/{id}/versions — تُعرض في UI
تعديلPUT title/description فقط
حذفمالك مشروع أو مالك الإصدار · يحذف file + DB
Push GitHubمالك المشروع + github_token → GithubService::pushVersionToGithub

GitHub OAuth

  1. المستخدم يضغط Link → redirect إلى GitHub (scope: repo).
  2. Callback stateless → token في users.github_token.
  3. Push يرسل محتوى الملف إلى repository المشروع.

10. سجل النشاط (Activity Log) — كيف استخدمناه

تبويب Activity log (ProjectTimeline.jsx) — ليس timeline للإصدارات.

  1. يستدعي GET /project/{id}/activities.
  2. يعرض project_activities (مهام، إصدارات، انضمام، …).
  3. فلتر: All · Tasks · Releases · Team.
  4. النص يُترجم عبر activityText.js + action_key.

يُكتب activity تلقائياً من Services عند: create task · status change · upload version · AI tasks · …

14. الذكاء الاصطناعي — ملخص

للشرح التفصيلي الكامل (prompts، Gemini API، regenerate، bookmarks، أخطاء، اختبارات، نقاط المناقشة): راجع PMS-AI-GUIDE.pdf — 14 فصلاً (~25 صفحة).

دمجنا Google Gemini (gemini-2.5-flash) كخدمة خارجية — مساعد للطالب وليس autopilot.

14.1 Project Ideation

ProjectIdeation.jsx → POST /ai/suggest-projects → AIIdeationService → 3 اقتراحات JSON · bookmarks (max 10) · regenerate مع exclude_names.

14.2 AI Task Generation

TasksTab.jsx → POST /projects/{id}/generate-tasks → AITaskService → 10 مهام · ai_generated=true · regenerate فقط إذا كل مهام AI لا تزال pending.

14.3 Adopt & Generate

من بطاقة الفكرة: إنشاء مشروع + توليد مهام + انتقال لتبويب Tasks في خطوة واحدة.

14.4 التكوين والأمان

GEMINI_API_KEY · config/ai.php · throttle: 10/h ideation · 10/min tasks · student only · sanitize + JSON validation.

15. التوطين i18n — كيف استخدمناه

  • ملفات en.js · ar.js — كل النصوص UI.
  • LanguageContext — dir RTL/LTR · lang للـ API (AI language instruction).
  • LanguageSwitcher في Layout.
  • Activity messages: action_key + activityText.js للترجمة الديناميكية.

ملخص للمناقشة

كيف نُقش استخدام المزايا:

  • كل feature = UI action + REST API + Service + Model — نمط موحّد.
  • Multi-tenant = TenantScope تلقائي — لا copy-paste filters.
  • Project hub = ProjectDetails tabs — كل collaboration في مكان واحد.
  • AI = external Gemini + config prompts + strict JSON parse — assistant وليس autopilot.
  • Notifications + Activity log = شفافية للفريق — من فعل ماذا ومتى.
