1. مقدمة وأهداف المعمارية

يهدف نظام PMS (Project Management System) إلى إدارة مشاريع التخرج الجامعية ضمن بيئة متعددة الجامعات (Multi-Tenant)، حيث تستضيف المنصة عدة جامعات كـ tenants مستقلين منطقياً، مع إمكانية إدارة مركزية على مستوى المنصة.

تسعى المعمارية إلى تحقيق الأهداف التالية:

  • فصل الاهتمامات (Separation of Concerns): فصل العرض عن منطق الأعمال عن البيانات.
  • قابلية الصيانة: تنظيم الكود في طبقات وخدمات (Services) يسهل تطويرها واختبارها.
  • عزل البيانات (Data Isolation): ضمان أن كل جامعة ترى بياناتها فقط.
  • مرونة الأدوار (RBAC): دعم super_admin، admin، supervisor، student بصلاحيات مختلفة.
  • قابلية التوسع التدريجي: بنية Monolith قابلة للفصل لاحقاً دون إعادة بناء كاملة.
  • دمج AI: ربط Google Gemini كخدمة خارجية دون coupling مباشر في Controllers.

5.3 إدارة الحالة (State Management)

لا يستخدم النظام Redux أو Zustand. تعتمد إدارة الحالة على:

  • AuthContext: المستخدم، Token، apiFetch، sessionBlock
  • LanguageContext: العربية/الإنجليزية + RTL
  • ThemeContext: Light/Dark + MUI Theme
  • Local State: useState في كل Page للبيانات المؤقتة

هذا اختيار مناسب لحجم المشروع الحالي — يقلل التعقيد مع الحفاظ على وضوح تدفق البيانات.

6. معمارية Multi-Tenancy

6.1 النموذج: Shared Database, Shared Schema

جميع الجامعات تستخدم قاعدة بيانات واحدة وجداول مشتركة. يتم العزل عبر عمود university_id وليس عبر قواعد منفصلة.

flowchart TB
    subgraph Platform
        SA[super_admin]
    end
    subgraph UniA[University A]
        ADM_A[admin A]
        USR_A[users A]
        PRJ_A[projects A]
    end
    subgraph UniB[University B]
        ADM_B[admin B]
        USR_B[users B]
        PRJ_B[projects B]
    end
    SA --> UniA
    SA --> UniB
    ADM_A --> USR_A --> PRJ_A
    ADM_B --> USR_B --> PRJ_B

6.2 TenantScope — آلية العزل

يُطبَّق TenantScope كـ Global Scope على نماذج Project و Task:

  • كل استعلام يُفلتر تلقائياً بـ university_id للمستخدم الحالي.
  • super_admin يتجاوز Scope ويرى كل الجامعات.
  • محاولة الوصول لـ ID من جامعة أخرى تُرجع 404 (وليس 403) — أمان أفضل.

6.3 Admin الجامعة — لا جدول منفصل

مدير الجامعة = سجل في users حيث roles.name = admin و users.university_id = معرف الجامعة. لا يوجد جدول admins.

7. معمارية المصادقة والأمان

7.1 Sanctum — Token-Based Authentication

sequenceDiagram
    participant C as Client React
    participant A as AuthController
    participant S as AuthService
    participant T as personal_access_tokens

    C->>A: POST /login
    A->>S: login credentials
    S->>T: createToken auth_token
    S-->>C: token + user
    C->>C: store in localStorage
    loop Every API call
        C->>A: Authorization Bearer token
        A->>A: auth:sanctum validates
    end

7.2 Role-Based Access Control (RBAC)

الدورالآليةالنطاق
super_adminrole middleware + bypass TenantScopePlatform-wide
adminrole middleware + university_id filterTenant
supervisorrole + ProjectAccessTenant + supervised projects
studentrole + ProjectAccess + AI routesTenant + owned/member projects

7.3 ProjectAccess — صلاحيات المشروع

بالإضافة إلى RBAC، يُستخدم ProjectAccess::canAccess() للتحقق من أن المستخدم مالك المشروع، مشرفاً عليه، أو عضواً مقبولاً — قبل أي عملية على مشروع محدد.

8. معمارية الذكاء الاصطناعي

8.1 Integration Pattern

يُدمج AI كـ External Service عبر Service Layer — Controllers لا تتصل بـ Gemini مباشرة.

flowchart LR
    FE[React UI] --> AC[AIIdeationController / AITaskController]
    AC --> SVC[AIIdeationService / AITaskService]
    SVC --> CFG[config/ai.php]
    SVC --> HTTP[HTTP Client]
    HTTP --> GEM[Google Gemini]
    SVC --> DB[(Logs + Tasks + Bookmarks)]

8.2 ميزتان AI

الميزةEndpointServiceالمخرجات
Project IdeationPOST /ai/suggest-projectsAIIdeationService3 أفكار + bookmarks
Task GenerationPOST /projects/{id}/generate-tasksAITaskService10 مهام ai_generated

8.3 حماية AI

  • Rate limiting: ideation 10/hr، tasks 10/min
  • Input sanitization قبل إرسال Gemini
  • JSON mode + schema validation للاستجابة
  • Logging في ideation_requests و task_generation_logs (بدون تخزين نص خام للاهتمامات)

14. القيود والتحسينات المستقبلية

قيود حالية:
  • Repository Pattern غير مكتمل — معظم Services تستعلم Eloquent مباشرة
  • لا Laravel Policies — الصلاحيات عبر Middleware + ProjectAccess
  • Queue = sync — لا background jobs للـ AI
  • Monolith — scaling أفقي محدود مقارنة بـ microservices

تحسينات مقترحة: Redis cache، Queue للـ AI requests، CI/CD، Repository كامل، Real-time notifications (WebSockets).

15. الخلاصة

يعتمد نظام PMS على معمارية Client–Server ثلاثية الطبقات مع Service Layer في Backend وComponent-Based SPA في Frontend. يُحقق عزل Multi-Tenant عبر Row-Level Isolation وTenantScope. المصادقة Stateless عبر Sanctum. يُدمج الذكاء الاصطناعي كخدمة خارجية دون تغيير البنية الأساسية. هذه المعمارية توازن بين البساطة وقابلية الصيانة ومتطلبات Multi-Tenant لمشاريع التخرج الجامعية.
