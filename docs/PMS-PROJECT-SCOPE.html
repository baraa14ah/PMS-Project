1. ملخص تنفيذي

PMS (Project Management System / ByteHub) منصة ويب لإدارة مشاريع التخرج والمشاريع الجامعية ضمن بيئة متعددة الجامعات (Multi-Tenant). يربط الطلاب والمشرفين ومديري الجامعات في مساحة عمل واحدة لكل institution، مع عزل بيانات كامل بين الجامعات.

النظام يوفّر: إنشاء المشاريع، بناء الفريق عبر الدعوات، إدارة المهام، نقاش الفريق (تعليقات المشروع)، رفع الإصدارات/الملفات، تكامل GitHub، إشعارات فورية، لوحات تحكم حسب الدور، وميزتين للذكاء الاصطناعي (اقتراح أفكار مشاريع + توليد مهام).

باختصار للمناقشة: PMS ليس مجرد todo list — هو منصة collaboration كاملة لمشاريع التخرج، مع tenant isolation، workflow موافقة حسابات، وصلاحيات على مستوى المشروع.

2. هدف المشروع ونطاقه

2.1 الهدف

  • تمكين الطلاب من إنشاء مشاريع التخرج وإدارة الفريق والمهام والملفات.
  • تمكين المشرفين من متابعة المشاريع وقبول/رفض دعوات الإشراف.
  • تمكين مدير الجامعة من إدارة المستخدمين والموافقة على الحسابات الجديدة.
  • تمكين مدير المنصة (Super Admin) من إدارة الجامعات والمنصة ككل.
  • دعم عدة جامعات على نفس المنصة دون تداخل بيانات.

4. البنية العامة وكيف يعمل النظام

4.1 النمط المعماري

  • Client–Server ثلاثي الطبقات: React SPA (Presentation) → Laravel REST API (Business) → MySQL (Data).
  • Monolith API + Service Layer (ProjectService, TaskService, AITaskService, …).
  • Shared Database Multi-Tenancy: جامعة واحدة = tenant؛ العزل عبر university_id + TenantScope.

4.2 تدفق عام للمستخدم

  1. يفتح التطبيق → يسجّل أو يدخل.
  2. إن كان الحساب pending → شاشة انتظار (الملف الشخصي + خروج فقط).
  3. إن كان active → لوحة تحكم حسب الدور.
  4. يعمل على المشاريع عبر: قائمة مشاريع → تفاصيل مشروع (تبويبات).
  5. كل عملية حساسة تمر عبر: Sanctum token + Role middleware + ProjectAccess.

5.2 إدارة المشاريع

  • إنشاء: طالب أو مدير جامعة · المشروع يرتبط تلقائياً بـ university_id.
  • الوصول: المالك · المشرف · الأعضاء المقبولون · مدير الجامعة · Super Admin.
  • التقدم: نسبة مئوية من المهام المكتملة.
  • GitHub: مزامنة commits · عرض سجل commits.
  • مغادرة الإشراف: المشرف يمكنه ترك المشروع.

5.3 الدعوات (Invitations)

  • دعوة مشرف: المالك يختار من قائمة مشرفين متاحين → المشرف يقبل/يرفض.
  • دعوة طالب: المالك يدعو طلاباً من نفس الجامعة → قبول/رفض → عضوية في project_members.

5.4 المهام (Tasks)

  • حالات: pending · in_progress · completed.
  • أي عضو بصلاحية وصول يمكنه إنشاء/تعديل/حذف.
  • تغيير الحالة يُسجَّل في Activity log ويُرسل إشعار.
  • نسبة الإنجاز تُحدَّث تلقائياً بعد كل عملية.

5.5 التعليقات

  • على مستوى المشروع فقط في واجهة المستخدم (team chat).
  • إضافة · عرض · تعديل · حذف (للمالك أو صاحب التعليق).

5.6 الإصدارات (Project Versions / Releases)

  • رفع ملف مع عنوان ووصف · تخزين في storage.
  • قائمة الإصدارات · تعديل metadata · حذف.
  • دفع الإصدار إلى GitHub (مالك المشروع + GitHub مربوط).

5.7 الإشعارات

  • تُنشأ عند: مهمة جديدة · تغيير حالة · رفع إصدار · دعوة · موافقة حساب · …
  • عرض · unread badge · mark read · delete.

6. خارج النطاق والقيود (Out of Scope)

  • لا WebSockets — الإشعارات polling/عند فتح الصفحة.
  • لا UI لتعليقات المهام (API موجود جزئياً في backend فقط).
  • Queue = sync — طلبات AI تنفَّذ synchronously.
  • لا Laravel Policies — الصلاحيات عبر Middleware + ProjectAccess helper.
  • لا نظام تقييم/درجات/ratings في التطبيق الحالي.
  • Version timeline API موجود لكن غير مستخدم في Frontend.

10. نقاط للمناقشة العامة

أسئلة شائعة في المناقشة:

  1. لماذا Multi-Tenant shared DB؟ — أبسط في النشر والصيانة لعدد جامعات محدود؛ العزل منطقي عبر university_id.
  2. لماذا Monolith وليس Microservices؟ — حجم المشروع لا يبرر تعقيد microservices.
  3. كيف يُعزل tenant؟ — TenantScope + university_id على كل استعلام tenant-scoped.
  4. Workflow المشرف multi-university؟ — pivot supervisor_universities · موافقة لكل جامعة · refreshAccountStatusFromMemberships.
  5. ProjectAccess vs Policies؟ — helper مركزي أبسط للمشروع الحالي؛ Policies أنسب للتوسع.
  6. دور AI؟ — مساعد ideation وتسريع task breakdown · ليس autopilot للمشروع.
  7. Activity log vs Version list؟ — Activity log = كل الأحداث · Releases tab = ملفات مرفوعة فقط.
  8. Scalability limits? — sync AI · no cache · monolith — مقبول للـ MVP/academic scope.
